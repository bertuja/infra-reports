⚠️
Watchers habilitados que no deberían estar en producción
El 94% del volumen es query + model + redis + cache. Estos watchers tienen valor en desarrollo pero son destructivos en producción con alto tráfico. Cada request de la API genera decenas de entradas en Aurora.
TelescopeServiceProvider.php — Estado actual (PROBLEMÁTICO) PHP
// Graba TODO en producción: Telescope::filter(function (IncomingEntry $entry) use ($isLocal) { return $isLocal || $entry->isReportableException() || $entry->isFailedRequest() || // ← 4xx/5xx masivos $entry->isFailedJob() || $entry->isScheduledTask() || $entry->hasMonitoredTag(); // ↑ Y con QueryWatcher, ModelWatcher, // RedisWatcher, CacheWatcher en TRUE // graba MILES de entradas por minuto });
TelescopeServiceProvider.php — Fix correcto PHP
// Solo errores críticos en producción: Telescope::filter(function (IncomingEntry $entry) use ($isLocal) { if ($isLocal) { return true; } // Producción: SOLO esto return $entry->isReportableException() || $entry->isFailedJob(); // REMOVIDO: isFailedRequest() → volumen masivo // REMOVIDO: isScheduledTask() → no crítico // REMOVIDO: query/model/redis/cache watchers });
.env — Watchers a deshabilitar en TODAS las apps de producción ENV
# Pegar en .env de producción — servijur_lv, interpost, milelogistic, space, crab TELESCOPE_QUERY_WATCHER=false TELESCOPE_MODEL_WATCHER=false TELESCOPE_REDIS_WATCHER=false TELESCOPE_CACHE_WATCHER=false TELESCOPE_EVENT_WATCHER=false TELESCOPE_VIEW_WATCHER=false TELESCOPE_GATE_WATCHER=false TELESCOPE_CLIENT_REQUEST_WATCHER=false # Después de editar el .env, ejecutar en cada app: # php artisan config:clear && php artisan config:cache
03
Plan de Limpieza — Borrado seguro por lotes
Ejecutar en DBeaver durante horario de bajo tráfico (2am–5am)
🚫
NUNCA ejecutar TRUNCATE directo en estas tablas
Con tablas de 100M+ filas un TRUNCATE o DELETE sin LIMIT genera un lock que mata la aplicación en producción, satura el binlog de Aurora y produce timeouts en cascada. Usar siempre DELETE ... LIMIT 10000.
1
ANTES DE EMPEZAR
Frenar Telescope en las 5 apps
Si no se frena primero, los datos vuelven a crecer mientras se borra. Aplicar el .env del paso anterior y limpiar config cache en cada instancia.
Ejecutar en cada instancia EC2 (SSH o SSM) BASH
# Para cada app: servijur_lv, interpost, milelogistic, space, crab cd /var/www/html/{app} # Agregar al .env echo "TELESCOPE_QUERY_WATCHER=false" >> .env echo "TELESCOPE_MODEL_WATCHER=false" >> .env echo "TELESCOPE_REDIS_WATCHER=false" >> .env echo "TELESCOPE_CACHE_WATCHER=false" >> .env # Aplicar cambios php artisan config:clear php artisan config:cache # Verificar que paró de grabar php artisan tinker --execute="echo app('Laravel\Telescope\Telescope')::isRecording() ? 'GRABANDO ⚠' : 'DETENIDO ✓';"
2
NOCHE — 2AM
Verificar rango de fechas antes de borrar
Ejecutar en DBeaver para confirmar fechas y cantidad real a borrar. Cambiar la base seleccionada en cada ejecución.
DBeaver — Ejecutar en cada base afectada SQL
USE servijur_lv; -- cambiar por: interpost, milelogistic, space, crab SELECT MIN(created_at) AS registro_mas_antiguo, MAX(created_at) AS registro_mas_nuevo, COUNT(*) AS total_filas, SUM(CASE WHEN created_at < NOW() - INTERVAL 7 DAY THEN 1 ELSE 0 END) AS filas_a_borrar, ROUND( SUM(CASE WHEN created_at < NOW() - INTERVAL 7 DAY THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 1 ) AS pct_a_borrar FROM telescope_entries;
3
NOCHE — LOTES DE 10K
Borrado por lotes — repetir hasta 0 filas afectadas
Ejecutar el DELETE repetidamente. Cuando devuelva menos de 10,000 filas afectadas, esa base está limpia. Pasar a la siguiente.
DBeaver — Script de borrado seguro (repetir en loop) SQL
USE servijur_lv; -- ORDEN: servijur_lv → interpost → milelogistic → space → crab -- [1] Cuánto falta borrar (ejecutar entre cada lote) SELECT COUNT(*) AS filas_restantes FROM telescope_entries WHERE created_at < NOW() - INTERVAL 7 DAY; -- [2] Borrado de 10,000 filas (repetir hasta que afecte < 10,000) DELETE FROM telescope_entries WHERE created_at < NOW() - INTERVAL 7 DAY LIMIT 10000; -- Tiempo estimado por base: -- servijur_lv → 94M filas → ~3-4 horas -- interpost → 101M filas → ~3-4 horas -- milelogistic → 66M filas → ~2-3 horas -- space → 28M filas → ~1 hora -- crab → 3.6M filas → ~15 minutos
4
POST LIMPIEZA
Configurar pruning automático
Agregar en app/Console/Kernel.php de cada app para que Telescope se auto-purgue diariamente. Con los watchers limitados, 48h de retención es más que suficiente.
app/Console/Kernel.php — Agregar schedule PHP
protected function schedule(Schedule $schedule): void { // Purgar Telescope diariamente — retener solo 48 horas $schedule->command('telescope:prune --hours=48')->daily(); // ... resto de schedules existentes }

Migración de Logs a S3 — Sacar los logs de Aurora
Plan para tablas loggers, logs, api_loggers (~180 GB). La solución más económica a largo plazo.

Logs en Aurora MySQL

✗ Crece sin límite — nunca se rotan
✗ Aurora cobra por GB almacenado
✗ Los backups automáticos incluyen los logs
✗ Queries lentas al buscar logs históricos
✗ Impacto en performance del cluster

Logs en archivos → S3
✓ S3 Standard: $0.023/GB/mes vs Aurora ~$0.10/GB

✓ Backups de Aurora más livianos y rápidos
✓ Cero impacto en queries de producción
✓ CloudWatch Logs para búsqueda y alertas

config/logging.php — Configuración correcta para producción PHP
// config/logging.php return [ 'default' => env('LOG_CHANNEL', 'daily'), // ← cambiar de 'stack' a 'daily' 'channels' => [ 'daily' => [ 'driver' => 'daily', 'path' => storage_path('logs/laravel.log'), 'level' => env('LOG_LEVEL', 'warning'), // ← solo warning+ en prd 'days' => 7, // ← retener 7 días en disco 'replace_placeholders' => true, ], ], ]; // En .env de producción: // LOG_CHANNEL=daily // LOG_LEVEL=warning ← no loguear debug/info en producción
Script bash — Sync diario de logs a S3 (cron en cada instancia) BASH
#!/bin/bash # /usr/local/bin/sync-logs-s3.sh # Cron: 0 3 * * * /usr/local/bin/sync-logs-s3.sh BUCKET="presis-logs-prod" APP_NAME="servijur_lv" # cambiar por app INSTANCE_ID=$(curl -s http://169.254.169.254/latest/meta-data/instance-id) DATE=$(date +%Y/%m/%d) LOG_PATH="/var/www/html/${APP_NAME}/storage/logs" # Sync logs del día anterior a S3 aws s3 sync $LOG_PATH s3://${BUCKET}/${APP_NAME}/${DATE}/${INSTANCE_ID}/ \ --exclude "*.log" \ --include "laravel-$(date -d yesterday +%Y-%m-%d).log" \ --storage-class STANDARD_IA echo "✓ Logs sincronizados: ${APP_NAME} → s3://${BUCKET}/${APP_NAME}/${DATE}/" # Eliminar logs de más de 7 días en disco (logrotate ya comprime) find $LOG_PATH -name "*.log.gz" -mtime +7 -delete
AWS CLI — Bucket S3 con lifecycle automático BASH
# Crear bucket aws s3api create-bucket \ --bucket presis-logs-prod \ --region us-east-1 # Lifecycle: Standard → Standard-IA (30d) → Glacier (60d) → Delete (365d) aws s3api put-bucket-lifecycle-configuration \ --bucket presis-logs-prod \ --lifecycle-configuration '{ "Rules": [{ "ID": "logs-lifecycle", "Status": "Enabled", "Filter": {"Prefix": ""}, "Transitions": [ {"Days": 30, "StorageClass": "STANDARD_IA"}, {"Days": 60, "StorageClass": "GLACIER"} ], "Expiration": {"Days": 365} }] }' # Costo estimado 1,000 GB de logs: # S3 Standard (0-30d): $23/mes # Standard-IA (30-60d): $12.50/mes # Glacier (60d+): $4/mes # vs Aurora actual: ~$100/mes por los mismos datos
